# Recipe: Svelte a11y warnings

## a11y_no_static_element_interactions — div with interactive events needs a role

### Problem

Svelte warns when a `<div>` has interactive event handlers (`ondrop`, `ondragover`, `onclick`, etc.) but no ARIA role. Assistive technologies can't identify the element's purpose.

### Fix

Add `role="region"` for a drop zone, or `role="button"` + `tabindex="0"` if the div is a clickable control.

```svelte
<!-- Drop zone: -->
<div
  role="region"
  aria-label="File drop zone"
  ondrop={handleDrop}
  ondragover={handleDragover}
>

<!-- Clickable control: -->
<div
  role="button"
  tabindex="0"
  onclick={handleClick}
  onkeydown={(e) => e.key === 'Enter' && handleClick()}
>
```

Screen readers announce elements by their role. A `<div>` with no role is announced as a generic container — users with assistive tech won't know it's interactive. The `onkeydown` handler on `role="button"` is required because `<div>` elements don't receive keyboard events by default.

## a11y_label_has_associated_control — label must be associated with its control

### Problem

`a11y_label_has_associated_control` fires when a `<label>` has no programmatic link to its input. Screen readers won't announce the label when the input is focused.

### Fix

Option A — `for`/`id` pair (prefer for layout flexibility):

```svelte
<label for="post-title">Title</label>
<input id="post-title" bind:value={title} />
```

Option B — wrap the input inside the label (no id needed, avoids duplicate-id issues when a component is rendered multiple times):

```svelte
<label>
  Title
  <input bind:value={title} />
</label>
```

The `for`/`id` link lets screen readers announce "Title, text field" when the input is focused. Without it the input is announced with no context.

## a11y_autofocus — autofocus attribute → programmatic focus via $effect

### Problem

Svelte warns `a11y_autofocus` on elements with the `autofocus` attribute. `autofocus` can disorient screen reader users by moving focus unexpectedly on page load.

### Fix

If focus-on-mount is genuinely needed (e.g. a modal's primary input), use a bound element reference and `$effect`:

```svelte
<script>
  let inputEl: HTMLInputElement | undefined = $state();
  $effect(() => { inputEl?.focus(); });
</script>
<input bind:this={inputEl} bind:value={val} />
```

If focus on mount is not critical, simply remove `autofocus`.

`$effect` runs after the component mounts and is explicit about the intent. It also gives you control over conditions (e.g. only focus when a modal opens, not on every re-render). The `autofocus` attribute fires immediately on HTML parse before the full page is ready, which can break screen reader virtual cursor position.
