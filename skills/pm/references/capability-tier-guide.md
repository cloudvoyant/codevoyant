# Capability Tier Guide

Reference for pm:plan. Describes how to assign features to tiers.

## What are capability tiers?

Capability tiers group features by strategic value and timing confidence — not by team, sprint, or component. They answer: "what must we ship, what should we ship if we can, and what are we only exploring?"

## Tier 1 — Core Capabilities

Features that are committed for the period. Must-have to achieve the strategic goal.

Criteria:
- Directly enables the stated strategic goal
- User or business outcome is well-understood
- Team has high confidence it can be delivered in the period

## Tier 2 — Growth Capabilities

High-value features that we intend to pursue, but are not committed.

Criteria:
- Advances the goal but not on the critical path
- Outcome understood but scope may shift
- Contingent on bandwidth or upstream completion

## Tier 3 — Future Capabilities

Features we are investing in lightly to validate direction. Not committed for this period.

Criteria:
- Strategic direction is right but timing uncertain
- Discovery or prototyping only
- Outcome hypothesis, not a confirmed bet

## Rules

- A capability should appear in only one tier
- Each capability must have: name, what it enables (outcome), why now (rationale), and 2–4 key bets
- No implementation details — roadmaps describe outcomes, not solutions
- No markdown tables — use definition lists or bullets
