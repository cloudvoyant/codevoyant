# pr

AI-powered code review — generate inline review comments, address change requests, and complete a draft review.

The `pr` skill orchestrates the full code review lifecycle: reading a PR/MR diff and generating professional, externally-referenced inline comments; pulling reviewer feedback and proposing targeted fixes; and publishing the completed review. It detects your provider (GitHub or GitLab) automatically and delegates to [`/gh`](/skills/gh) or [`/glab`](/skills/glab) for all platform operations.


## Installation

**Claude Code:**
```bash
npx skills add cloudvoyant/codevoyant
```

**OpenCode / VS Code Copilot:** See the [installation guide](/user-guide#installation).

## Requirements

- [`gh` CLI](https://cli.github.com/) for GitHub projects (`gh auth login`)
- [`glab` CLI](https://gitlab.com/gitlab-org/cli) for GitLab projects (`glab auth login`)
- Active Git repository with a GitHub or GitLab remote

## Typical Workflows

### Review an open PR/MR

```bash
/pr new                   # auto-detects PR/MR from current branch
/pr new 42                # priew a specific PR/MR number
```

The skill reads the full diff and generates inline comments. When done, it asks whether to submit as a draft or publish immediately.

### Address review comments on your PR/MR

```bash
/pr address               # pull comments and propose fixes for current branch
/pr address 42            # address a specific PR/MR
```

Fetches unresolved threads, proposes file-level fixes one-by-one, lets you accept or skip each, then applies approved changes and posts draft responses to each thread.

### Publish a pending draft review

```bash
/pr complete
/pr complete --event APPROVE             # approve
/pr complete --event REQUEST_CHANGES     # request changes
```

### Name your review documents

```bash
/pr new --name auth-refactor       # saves to .codevoyant/review/auth-refactor/
/pr address --name auth-refactor   # picks up the same session
```

## Skills

### pr new — Generate Review

Read a PR/MR diff and generate AI-authored inline comments:

```bash
/pr new [pr-id] [--name <slug>] [--github|--gitlab]
```

**Review quality rules — non-negotiable:**
- Professional and courteous — no sarcasm, no rhetorical questions, no faint praise
- Terse — one sentence where one sentence suffices
- Constructive — every comment tells the author exactly what to change
- Proposes concrete code changes (diff blocks) for non-trivial issues
- Cites relevant docs, RFCs, or prior art where it strengthens the point
- Skips conforming code — does not nitpick style that matches project conventions

**Severity levels:**
- `MUST CHANGE` — bug, security issue, or logic error (blocking)
- `CONSIDER` — style, naming, structure (non-blocking)
- `NOTE` — informational observation

**What happens:**
1. Detects provider from git remote URL
2. Detects open PR/MR from current branch (or uses provided number)
3. Fetches the diff via `gh pr diff` / `glab mr diff`
4. Generates a JSON array of inline comments with an Opus-class review agent
5. Writes the review to `.codevoyant/review/{slug}/new-review.md`
6. Asks: submit as draft, submit and publish, or save for later

**Review document:** `.codevoyant/review/{slug}/new-review.md`

### pr address — Address Review Comments

Pull open review threads and work through them interactively:

```bash
/pr address [pr-id] [--name <slug>] [--github|--gitlab]
```

**What happens:**
1. Fetches unresolved review threads via `/gh pull-comments` or `/glab pull-comments`
2. Writes threads to `.codevoyant/review/{slug}/comments.md`
3. For each thread, proposes a concrete fix with a diff block
4. Asks: apply all fixes, or mark some `SKIP` first and re-run
5. Applies approved fixes by direct file edit (no `patch` command)
6. Posts a draft response to each addressed thread

To skip specific fixes before applying: open `address.md`, change `**Status:** PENDING` to `**Status:** SKIP` for the fixes you want to exclude, then rerun `/pr address`.

**Review document:** `.codevoyant/review/{slug}/address.md`

### pr complete — Publish Draft Review

Publish a pending draft review or draft MR:

```bash
/pr complete [pr-id] [--event APPROVE|REQUEST_CHANGES|COMMENT] [--github|--gitlab]
```

Delegates to `/gh complete` or `/glab complete`. For GitHub, validates that a pending draft review exists first.

### Review Document Structure

All review artifacts are stored under `.codevoyant/review/`:

```
.codevoyant/review/
  {slug}/
    new-review.md    ← generated by rev new
    comments.md      ← fetched by rev address
    address.md       ← fix proposals from rev address
```

The slug defaults to a normalized version of the PR/MR title (`my-feature-name`). Pass `--name` to set it explicitly and share documents across `new`, `address`, and `complete` invocations.
