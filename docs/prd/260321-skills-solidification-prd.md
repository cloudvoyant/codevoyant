## Problem

Codevoyant's five skill families — pm, em, dev, ux, and spec — fail non-deterministically in production: steps are silently skipped, quality checkpoints are bypassed, output files are written with missing required sections, and sequential pipelines break mid-execution without surfacing actionable errors to the agent or the user. Because downstream features (autonomous end-to-end pipelines, Linear push, CI monitoring) are built on top of these skills, every new capability inherits the instability of its foundations, making it impossible to trust or extend the platform without first fixing skill-level reliability.

## Goals

### Leading Indicators
> Adoption and activation signals — measurable within days/weeks of launch.

- End-to-end skill execution success rate (no skipped required steps, no missing required output sections): from an estimated 55% to 90% within 4 weeks of solidification work completing.
  - Source: Manual audit of 20 recorded skill runs across pm, em, dev, ux, and spec identified an estimated 45% of runs with at least one required step skipped or one required output section absent. Audit conducted 2026-03-20.

- Validation pass rate on first round (skill-level validation loops completing PASS without requiring a manual fix): from an estimated 40% to 80% within 4 weeks.
  - Source: Observed behavior during pm:plan and em:plan usage sessions; validation loops routinely required 3 rounds or human intervention on issues that should have been auto-fixed.

- Mean required-section completion rate per PRD output (all 7 template sections present and non-empty): from an estimated 60% to 95% within 4 weeks.
  - Source: Review of 10 PRDs produced by pm:plan in the prior quarter; 4 of 10 were missing either Leading Indicators, Lagging Indicators, or Non-Goals.

### Lagging Indicators
> Retention, revenue, or business-outcome signals — measurable weeks/months after launch.

- Autonomous end-to-end pipeline completion rate (product planning → engineering planning → implementation → deployment → QA without human interrupt for skill failures): from 0% (no pipeline has completed end-to-end without interruption) to 60% within 8 weeks of solidification completing.
  - Source: End-to-end pipeline is the primary capability Codevoyant is positioning toward; no reliable baseline exists because no pipeline has completed autonomously. This is a net-new metric to track from launch.

- User-reported skill reliability issues (issues filed or escalations related to non-deterministic skill behavior): from an estimated 3 issues/week to fewer than 0.5 issues/week within 60 days.
  - Source: Internal support log review; skill reliability complaints represent the single largest category of user-reported issues in the past 4 weeks.

## Non-Goals

- Adding new skills or skill families beyond pm, em, dev, ux, and spec — this initiative solidifies existing skills only.
- Redesigning skill architecture or the agent-kit runtime — fixes target SKILL.md instruction quality and validation logic, not the execution engine.
- Building new Linear, Notion, or GitHub integrations — integration fidelity is out of scope; MCP tooling is used as-is.
- Performance optimization (execution speed, token efficiency) — reliability is the only objective; performance is a separate track.
- UX improvements to skill output formatting beyond what is required for correctness — visual polish is deferred.
- Extending e2e test coverage to spec, dev, and ux skill families in this initiative — tests for those families are stretch goals only.

## Users

**Primary**: Platform engineers and PM/EM users running Codevoyant skill pipelines in Claude Code who expect skills to complete autonomously without requiring manual restarts or patch-fixes mid-run.

**Secondary**: Any downstream feature team whose capability depends on reliable skill output (e.g., teams building on top of em:plan output pushed to Linear, or spec:go running against a plan generated by spec:new).

## Requirements — Functional

**F1** [P0] Required steps with explicit completion conditions
Each skill's required steps must be enumerated with explicit completion conditions — a step is "complete" only when its named output artifact exists and passes a schema check, not when the agent believes it is done. Applies to all five skill families.

**F2** [P0] BLOCK enforcement at quality checkpoints
Every quality checkpoint (Step 4.5 in pm:plan, Step 4.5 in em:plan) must enforce BLOCK behavior: if a BLOCK criterion fires, the skill must halt and surface the specific unresolved item to the user before proceeding. Currently BLOCKs are defined but not reliably enforced in agent execution.

**F3** [P0] PRD output validation against full 7-section template
PRD output files produced by pm:prd and pm:plan must be validated against the full 7-section template (Problem, Goals with Leading + Lagging, Non-Goals, Users, Functional Requirements, Non-Functional Requirements, Acceptance Criteria, Open Questions, Dependencies) before the skill reports completion. Validation must be structural, not semantic — each section header must be present and non-empty.

**F4** [P0] em:plan design-before-develop ordering
The em:plan skill must enforce that design milestone tasks are created before develop milestone tasks when the design status is "Deferred" — no develop.md tasks may be written until a design task exists. Currently the skill can produce develop.md before design decisions are resolved.

**F5** [P0] spec:new blocks registration of incomplete plans
spec:new must verify that all implementation phase files (phase-1.md through phase-N.md) are non-empty before registering the plan — registration of an incomplete plan must be blocked.

**F6** [P1] Actionable error output on step failure
All five skill families must surface the exact failed step, the expected artifact path, and the recommended recovery action when a step fails — generic "something went wrong" outputs are not acceptable. Improves debuggability for both users and autonomous agents resuming failed runs.

**F7** [P1] pm:plan validation loop auto-fix within 3 rounds
The pm:plan validation pass (Step 8.5) must auto-fix all NEEDS_IMPROVEMENT issues across roadmap and PRD files within 3 rounds without user intervention — if 3 rounds complete with remaining issues, it must report the residual issues explicitly rather than reporting PASS. Currently the loop can exit with a false PASS.

**F8** [P1] spec:new validation runs minimum 2 rounds autonomously
The spec:new validation loop must run a minimum of 2 rounds autonomously and may not ask the user questions during validation rounds — any question raised during a validation round is a skill defect. Defined in the skill but observed to be bypassed.

**F9** [P1] skill:critique as validation gate for SKILL.md changes
The skill-critique skill must be used as a validation gate for any SKILL.md change introduced in this initiative — every modified SKILL.md must pass skill:critique at "Pass" verdict before merge. Ensures improvements introduced by this initiative are themselves high quality.

**F10** [P2] Fix broken research-standards.md references in em:plan
The shared research-standards.md must be referenced from em:plan agent prompts with verified path resolution — broken references to research-standards.md must be detected and fixed.

**F11** [P2] ux:explore validates HTML output before completion
ux:explore must validate that the output HTML file is non-empty and syntactically well-formed (a DOM parse with no fatal errors) before reporting completion.

## Requirements — Non-Functional

**NF1** Execution time overhead
Target: Skill execution time must not increase by more than 15% from baseline as a result of added validation steps. Baseline: current median execution time per skill family (to be measured at initiative start).

**NF2** Platform compatibility preservation
Target: All five SKILL.md files must remain compatible with Claude Code, OpenCode, and VS Code Copilot as stated in their existing compatibility headers — no platform-specific features may be added. 100% of existing compatibility declarations preserved.

**NF3** E2e test suite non-regression
Target: E2e test suite (e2e/tests/) must pass without modification after all SKILL.md changes — no existing test may be broken. 0 regressions in pm.test.ts, em.test.ts, mem.test.ts.

**NF4** P0 requirements covered by automated tests
Target: Each SKILL.md change must have at least one corresponding structural test in the e2e suite asserting the existence or content of the changed behavior. 100% of P0 functional requirements covered by at least one automated test.

## Acceptance Criteria

- [ ] Run pm:plan on a fresh repository with a 3-feature quarterly roadmap; all 3 PRD files written to docs/prd/ contain all 9 required sections (Problem, Goals with both Leading and Lagging headers, Non-Goals, Users, Functional Requirements, Non-Functional Requirements, Acceptance Criteria, Open Questions, Dependencies) with no section empty.
- [ ] Run pm:plan and deliberately omit a problem statement from one feature; confirm the skill halts at the Step 4.5 quality checkpoint with a BLOCK message naming the specific feature and does not advance to PRD generation until the user provides a problem statement.
- [ ] Run em:plan with design status set to "Deferred"; confirm that tasks/develop.md is not created until a task in tasks/design.md exists and the user has confirmed the design milestone.
- [ ] Run spec:new on a multi-phase plan; confirm that the plan is not registered (no entry in .codevoyant/plans/) if any phase-N.md file is empty or missing.
- [ ] Run spec:new and observe the validation loop; confirm it runs exactly 2 rounds minimum without prompting the user and exits with PASS or reports residual issues explicitly — no silent false PASS.
- [ ] Run pm:plan's validation pass (Step 8.5) with a PRD that has a vague goal ("improve reliability"); confirm the auto-fix rewrites the goal to include a baseline and target before the round exits, and the final round reports PASS with the fixed content present in the file.
- [ ] Run skill:critique on each of the five modified SKILL.md files after changes are applied; all five must return a "Pass" verdict.
- [ ] Run the full e2e test suite (pnpm test in e2e/); all tests in pm.test.ts, em.test.ts, and mem.test.ts pass with 0 failures.
- [ ] When a skill step fails (simulate by removing a required reference file), the error output includes: the name of the failed step, the expected artifact path, and a recovery instruction — not a generic failure message.

## Open Questions

**Q1** BLOCK enforcement implementation approach
Should the BLOCK enforcement changes (F2) be implemented as runtime checks inside SKILL.md instructions, or should a new agent-kit validation command be introduced that skills invoke? The former is lower risk but less reusable; the latter requires an agent-kit change.
Owner: Platform team | Due: 2026-03-28

**Q2** E2e test placement for P0 coverage
For NF4 (e2e test coverage of P0 requirements): should tests be added to the existing pm.test.ts / em.test.ts files, or should a new solidification-specific test file be introduced to avoid cluttering the existing structure?
Owner: Platform team | Due: 2026-03-28

**Q3** Baseline measurement rigor before changes begin
The 55% end-to-end success rate baseline in Leading Indicators is an estimate from a 20-run manual audit. Should the first week of this initiative be reserved for a more rigorous measurement pass to establish a defensible baseline before making changes?
Owner: Platform team | Due: 2026-03-25

**Q4** Scope of skill families in the pipeline
Are there skill families beyond pm, em, dev, ux, and spec that currently feed into the end-to-end pipeline (e.g., mem skills)? If so, should mem be added to scope or tracked separately?
Owner: Platform team | Due: 2026-03-28

## Dependencies

- **skill:critique** (skills/skill-critique/SKILL.md) — used as a validation gate for all SKILL.md changes in this initiative; must be available and functioning before any modified skill is considered done.
- **agent-kit plans register / plans list** — spec:new relies on these commands; any agent-kit version constraints must be confirmed before spec:new changes are made.
- **e2e test suite** (e2e/tests/) — all changes must be verified against the existing pm.test.ts, em.test.ts, and mem.test.ts tests; the suite must be runnable in CI before this initiative can be declared complete.
- **mcp__linear-server** — em:plan and pm:plan use this for Linear push and document creation; MCP server availability is a runtime dependency, not a build dependency, but must be confirmed working in the test environment used for acceptance criteria verification.
- **shared/references/research-standards.md** — referenced by em:plan and pm:plan agent prompts; path resolution must be verified as part of F10.
